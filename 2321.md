# 日报/周报 Dashboard 数据库设计方案（V2）

## 1. 背景

本 Dashboard 用于展示企业内部日报/周报相关数据，主要包含以下模块：

1. 数据监控
   - 每个部门已填写人数 / 未填写人数
   - 一周填写率排名
   - 部门过去一周日报填写率趋势

2. 平台项目日报卡片
   - 平台项目日报关键进展
   - 每个平台项目的 AI 日报链接
   - 每个平台项目的原文链接

3. 部门日报卡片
   - 部门日报关键进展
   - 每个部门的 AI 日报链接
   - 每个部门的原文链接，如有

4. 平台项目周报卡片
   - 平台项目周报关键进展
   - 每个平台项目的 AI 周报链接

5. 部门周报卡片
   - 部门周报关键进展
   - 每个部门的 AI 周报链接

6. 困难及风险
   - 每个叶子部门的每一条困难及风险原文
   - 提出人
   - 关联的平台项目
   - 关联的部门项目
   - 风险等级，预留字段

7. 下一步计划
   - 每个叶子部门的每一条下一步计划原文
   - 提出人
   - 关联的平台项目
   - 关联的部门项目

---

## 2. 当前日报结构说明

当前只要求叶子部门填写原始日报。

组织结构大致为：

```text
L3部门
 └── L4部门
      └── L5部门
```

通常情况下：

```text
L5部门 = 叶子部门 = 原始日报填写部门
```

每天每个叶子部门一篇原始日报。

原始日报内容按照人员维度填写，例如：

```text
@张三
- 项目A进展
- 项目B风险
- 下一步计划

@李四
- 项目C进展
- 下一步计划
```

---

## 3. 原子池抽取说明

当前已有一个原子池应用，负责把原始日报内容结构化抽取为 JSON。

抽取内容包括：

```text
meta 信息
人 -> 项目
平台 -> 项目 -> 进展 / 困难 / 下一步计划
```

需要注意：

```text
原子池只抽取有实际进展的人
```

因此：

```text
原子池中出现的人 ≠ 部门实际人员
```

如果要做填写率统计，需要将：

```text
部门实际人员列表
```

和：

```text
原子池抽取出的人
```

进行对比。

---

## 4. 设计原则

本版本优先服务 BI 看板展示，因此采用较简化的数据模型。

核心原则：

```text
事实表只存当前归属部门 dept_id
部门层级关系统一通过 dim_department 解析
```

也就是说，事实表中不强制存：

```text
l3_dept_id
l4_dept_id
l5_dept_id
parent_dept_id
```

而是统一存：

```text
dept_id
```

然后通过 `dim_department` 的 `parent_dept_id` 和 `dept_path` 解析上级部门。

这样可以减少数据写入时的复杂度。

# 7. 部门维表

## 7.1 表名

```text
dim_department
```

## 7.2 说明

用于存储 L3 / L4 / L5 部门层级关系。

虽然事实表只存 `dept_id`，但 BI 可以通过该表查询任意部门的上级、下级和完整路径。

---

## 7.3 字段设计

| 字段 | 类型 | 说明 |
|---|---|---|
| dept_id | VARCHAR(64) | 部门 ID，主键 |
| dept_name | VARCHAR(255) | 部门名称 |
| dept_level | INT | 部门层级，例如 3 / 4 / 5 |
| parent_dept_id | VARCHAR(64) | 上级部门 ID |
| dept_path | VARCHAR(1000) | 部门路径，例如 `/L3_ID/L4_ID/L5_ID` |
| is_leaf | BOOLEAN | 是否叶子部门 |
| is_active | BOOLEAN | 是否有效 |

---

## 7.4 建表示例

```sql
CREATE TABLE dim_department (
    dept_id          VARCHAR(64) PRIMARY KEY,
    dept_name        VARCHAR(255),
    dept_level       INT,
    parent_dept_id   VARCHAR(64),
    dept_path        VARCHAR(1000),
    is_leaf          BOOLEAN,
    is_active        BOOLEAN
);
```

---

## 7.5 示例数据

| dept_id | dept_name | dept_level | parent_dept_id | dept_path | is_leaf |
|---|---|---:|---|---|---|
| D_L3_001 | 研发中心 | 3 | NULL | /D_L3_001 | false |
| D_L4_001 | 平台研发部 | 4 | D_L3_001 | /D_L3_001/D_L4_001 | false |
| D_L5_001 | AI 平台组 | 5 | D_L4_001 | /D_L3_001/D_L4_001/D_L5_001 | true |

# 10. 日报填写覆盖事实表

## 10.1 表名

```text
fact_daily_user_coverage
```

## 10.2 说明

用于统计每个人每天是否填写日报。

这张表用于支持：

```text
每个部门已填写人数 / 未填写人数
一周填写率排名
部门过去一周日报填写率趋势
```

---

## 10.3 生成逻辑

每天针对每个叶子部门：

1. 从 `dim_user` 获取该部门实际人员列表。
2. 从原始日报中解析出现过的人。
3. 从原子池 JSON 中获取被抽取出的人。
4. 生成人员覆盖记录。

判断逻辑：

```text
部门实际人员 - 原子池抽取人员 = 未填写或无有效进展人员
```

---

## 10.4 字段设计

| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGINT | 主键 |
| date_id | DATE | 日期 |
| user_guid | VARCHAR(64) | 用户 ID |
| dept_id | VARCHAR(64) | 用户所属叶子部门 ID |
| should_submit | BOOLEAN | 是否应填写 |
| appeared_in_raw | BOOLEAN | 是否在原始日报中出现 |
| extracted_by_atom | BOOLEAN | 是否被原子池抽取 |
| submission_status | VARCHAR(32) | 填写状态 |
| raw_daily_doc_url | TEXT | 原始日报链接 |
| created_at | DATETIME | 创建时间 |

---

## 10.5 建表示例

```sql
CREATE TABLE fact_daily_user_coverage (
    id                  BIGINT PRIMARY KEY AUTO_INCREMENT,
    date_id             DATE,
    user_guid           VARCHAR(64),
    dept_id             VARCHAR(64),

    should_submit       BOOLEAN,
    appeared_in_raw     BOOLEAN,
    extracted_by_atom   BOOLEAN,

    submission_status   VARCHAR(32),

    raw_daily_doc_url   TEXT,
    created_at          DATETIME
);
```

---

## 10.6 submission_status 说明

| 状态 | 含义 |
|---|---|
| submitted | 已填写，并被原子池抽取到有效内容 |
| not_submitted | 未在原始日报中出现，也未被原子池抽取 |
| empty_content | 在原始日报中出现，但没有有效进展 |
| parse_failed | 原始日报或原子池解析失败 |

---

## 10.7 指标计算

### 已填写人数

```sql
COUNT(DISTINCT user_guid)
WHERE submission_status = 'submitted'
```

### 未填写人数

```sql
COUNT(DISTINCT user_guid)
WHERE submission_status <> 'submitted'
```

### 填写率

```text
已填写人数 / 应填写人数
```

其中：

```text
应填写人数 = should_submit = true 的人数
```

# 14. BI 筛选逻辑

## 14.1 选择 L5 部门

如果用户选择的是 L5 部门：

```sql
WHERE fact_table.dept_id = selected_dept_id
```

适用于：

```text
日报填写覆盖
部门日报
部门周报
困难风险
下一步计划
```

填写率排名展示：

```text
该 L5 部门下每个人的填写率排名
```

---

## 14.2 选择 L4 部门

如果用户选择的是 L4 部门，需要先找到其下所有叶子部门。

```sql
SELECT dept_id
FROM dim_department
WHERE parent_dept_id = selected_l4_dept_id
  AND is_leaf = TRUE;
```

然后：

```sql
WHERE fact_table.dept_id IN (...)
```

填写率排名展示：

```text
该 L4 部门下所有 L5 部门的填写率排名
```

---

## 14.3 选择 L3 部门

如果用户选择的是 L3 部门，需要找到其下所有叶子部门。

方式一：递归查询。

方式二：使用 `dept_path`。

推荐使用 `dept_path`：

```sql
SELECT dept_id
FROM dim_department
WHERE dept_path LIKE CONCAT('%/', selected_l3_dept_id, '/%')
  AND is_leaf = TRUE;
```

然后：

```sql
WHERE fact_table.dept_id IN (...)
```

填写率排名展示：

```text
该 L3 部门下所有 L4 或 L5 部门的填写率排名
```

---

## 14.4 平台项目卡片

平台项目卡片不受部门筛选影响。

查询条件：

```sql
WHERE report_scope = 'platform_project'
```

日报：

```sql
WHERE report_scope = 'platform_project'
  AND report_type = 'daily'
```

周报：

```sql
WHERE report_scope = 'platform_project'
  AND report_type = 'weekly'
```

# 15. 常用查询示例

## 15.1 某 L5 部门当日填写情况

```sql
SELECT
    f.date_id,
    d.dept_name,
    COUNT(CASE WHEN f.should_submit = TRUE THEN 1 END) AS should_submit_count,
    COUNT(CASE WHEN f.submission_status = 'submitted' THEN 1 END) AS submitted_count,
    COUNT(CASE WHEN f.submission_status <> 'submitted' THEN 1 END) AS not_submitted_count
FROM fact_daily_user_coverage f
JOIN dim_department d
  ON f.dept_id = d.dept_id
WHERE f.date_id = '2026-06-10'
  AND f.dept_id = 'D_L5_001'
GROUP BY f.date_id, d.dept_name;
```

---

## 15.2 某 L4 部门下属 L5 一周填写率排名

```sql
SELECT
    leaf.dept_id,
    leaf.dept_name,
    COUNT(CASE WHEN f.should_submit = TRUE THEN 1 END) AS should_submit_count,
    COUNT(CASE WHEN f.submission_status = 'submitted' THEN 1 END) AS submitted_count,
    ROUND(
        COUNT(CASE WHEN f.submission_status = 'submitted' THEN 1 END) * 1.0 /
        NULLIF(COUNT(CASE WHEN f.should_submit = TRUE THEN 1 END), 0),
        4
    ) AS submission_rate
FROM fact_daily_user_coverage f
JOIN dim_department leaf
  ON f.dept_id = leaf.dept_id
WHERE leaf.parent_dept_id = 'D_L4_001'
  AND f.date_id BETWEEN '2026-06-03' AND '2026-06-10'
GROUP BY leaf.dept_id, leaf.dept_name
ORDER BY submission_rate DESC;
```

---

## 15.3 某 L5 部门人员一周填写率排名

```sql
SELECT
    u.user_guid,
    u.user_name,
    COUNT(CASE WHEN f.should_submit = TRUE THEN 1 END) AS should_submit_count,
    COUNT(CASE WHEN f.submission_status = 'submitted' THEN 1 END) AS submitted_count,
    ROUND(
        COUNT(CASE WHEN f.submission_status = 'submitted' THEN 1 END) * 1.0 /
        NULLIF(COUNT(CASE WHEN f.should_submit = TRUE THEN 1 END), 0),
        4
    ) AS submission_rate
FROM fact_daily_user_coverage f
JOIN dim_user u
  ON f.user_guid = u.user_guid
WHERE f.dept_id = 'D_L5_001'
  AND f.date_id BETWEEN '2026-06-03' AND '2026-06-10'
GROUP BY u.user_guid, u.user_name
ORDER BY submission_rate DESC;
```

```sql
FROM fact_next_plan_item n
LEFT JOIN dim_department d
  ON n.dept_id = d.dept_id
LEFT JOIN dim_platform_project p
  ON n.platform_project_id = p.project_id
WHERE n.date_id = '2026-06-10'
  AND n.dept_id = 'D_L5_001';
```
