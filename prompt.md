# 更好地使用 Chatbot：通用提示词入门指南

## 1. 什么是提示词

提示词，英文通常称为 **Prompt**，指的是我们输入给 Chatbot 的内容。

它可以是一句话，例如：

> 帮我总结这篇文章。

也可以是一段包含背景、任务、材料和要求的完整说明，例如：

> 我正在准备一次项目汇报，请帮我把下面的会议记录整理成一份 300 字以内的摘要，重点保留项目进展、当前问题和下一步计划。

因此，提示词并不只是一个问题，也可以包括：

- 任务背景
- 需要模型处理的材料
- 希望模型完成的动作
- 输出格式
- 内容重点
- 长度和语气要求
- 需要遵守的规则
- 对结果的修改意见

从使用者的角度看，提示词可以理解为：

> **我们向 Chatbot 描述任务、传递信息和提出要求的方式。**

---

## 2. Chatbot 是怎么理解我们的输入的

Chatbot 并不是像人一样真正经历过项目，也不会自动知道我们脑中的背景。

它主要根据当前对话中出现的文字，判断：

1. 你正在讨论什么
2. 你希望它完成什么
3. 哪些信息是重要的
4. 输出应该采用什么形式
5. 前面的对话是否需要继续参考

例如，我们输入：

> 帮我写一下。

虽然人可能知道“写一下”指什么，但对 Chatbot 来说，这句话包含的信息很少。

它不知道：

- 要写什么
- 写给谁看
- 用在什么场景
- 需要多长
- 使用什么语气
- 重点是什么
- 是否有现成材料

因此，模型只能根据有限的信息进行推测。

如果我们改成：

> 请把下面的项目记录整理成一封发给部门负责人的邮件。邮件需要简洁、正式，重点说明项目延期原因和需要协调的资源，控制在 300 字以内。

模型可以识别出更多信息：

| 信息类型 | 输入内容 |
|---|---|
| 任务 | 整理成一封邮件 |
| 读者 | 部门负责人 |
| 语气 | 简洁、正式 |
| 重点 | 延期原因、需要协调的资源 |
| 长度 | 300 字以内 |
| 材料 | 后面提供的项目记录 |

输入越明确，模型需要自行猜测的内容越少，输出通常也越接近我们的需求。

## 3. 模型不是只看关键词，而是在理解上下文

Chatbot 在生成回答时，并不是简单地查找某个关键词对应的固定答案。

它会综合参考：

- 当前问题
- 前后文
- 用户提供的材料
- 任务描述
- 输出要求
- 对话中之前提出的修改意见

例如：

> 请帮我整理下面的内容。

模型知道需要“整理”，但不知道整理成什么。

如果继续补充：

> 请整理成表格。

模型就会调整输出形式。

再补充：

> 表格包括事项、负责人、截止时间和当前状态。

模型会进一步按照指定字段组织信息。

这说明和 Chatbot 沟通通常是一个逐步明确任务的过程。

```text
第一次输入：说明大致任务
第二次输入：补充背景或材料
第三次输入：调整结构和重点
第四次输入：修改语言和格式
```

因此，使用 Chatbot 不一定要追求一次就写出一个非常复杂的提示词。

更实用的方法是：

> **先开始，再通过对话逐步补充和修正。**

## 4. Chatbot 如何生成回答

为了更好地理解提示词的作用，可以把 Chatbot 的工作过程简单理解为以下几步。

### 4.1 识别任务

模型首先判断用户希望它做什么。

常见任务包括：

- 总结
- 翻译
- 改写
- 分析
- 对比
- 解释
- 生成
- 提取
- 分类
- 检查
- 制定计划

例如：

> 请比较方案 A 和方案 B。

这里的核心任务是“比较”。

### 4.2 提取上下文

模型会从输入中提取完成任务需要的信息。

例如：

> 我们是一个 5 人团队，需要在一个月内上线内部工具，请比较方案 A 和方案 B。

模型会识别到：

- 团队规模是 5 人
- 时间限制是一个月
- 使用场景是内部工具
- 需要比较两个方案

这些信息会影响它判断哪个方案更合适。

### 4.3 识别要求和限制

模型还会关注输出应该满足什么要求。

例如：

> 请从成本、开发周期、学习成本和维护难度四个维度比较，并使用表格输出。

这里规定了：

- 比较维度
- 输出结构

如果没有这些要求，模型可能自行选择比较角度，结果不一定符合使用者的真正需要。

### 4.4 生成回答

模型会根据任务、上下文和要求，生成一个最可能符合输入意图的回答。

可以把这个过程简单理解为：

```text
模型输出
=
任务理解
+ 上下文信息
+ 具体要求
+ 对话历史
```

因此，提示词的主要作用不是使用某种特殊句式，而是帮助模型更准确地理解任务。

## 5. 为什么同一个问题会得到不同质量的回答

### 提问方式一

> 帮我写一个项目总结。

这个提示词只告诉模型需要“写项目总结”。

但没有说明：

- 项目内容
- 当前阶段
- 总结目的
- 阅读对象
- 内容重点
- 篇幅
- 输出结构

模型只能生成一份比较通用的内容。

### 提问方式二

> 请帮我写一份项目阶段总结。
>
> 项目是一个内部知识管理工具，目前已经完成第一阶段试用。
>
> 总结将发给部门负责人，目的是汇报当前进展，并申请下一阶段资源。
>
> 已有信息：
>
> - 完成了需求调研
> - 完成了第一版原型
> - 20 名员工参与试用
> - 用户反馈主要集中在搜索结果不够准确
> - 下一阶段计划优化搜索功能并扩大试用范围
>
> 请按照“项目背景、当前进展、存在问题、下一步计划”四部分组织，控制在 500 字以内，使用简洁、正式的语言。

第二种提问提供了完整的任务信息，因此输出通常会更加具体。

两种提问的差异不在于是否使用了复杂技巧，而在于是否回答了以下问题：

```text
要做什么？
为什么要做？
基于什么材料？
给谁使用？
重点是什么？
输出成什么样？
```

## 6. 一个通用的提示词结构

一个比较完整的提示词通常可以包括五个部分：

```text
背景
任务
材料
要求
补充说明
```

不需要每次都使用全部内容，可以根据任务复杂度进行选择。

### 6.1 背景

背景用于说明当前场景，帮助模型理解任务为什么存在。

例如：

> 我正在准备一场面向新员工的培训。

> 我需要向部门负责人汇报项目进度。

> 我正在分析一次用户满意度调查。

> 我准备回复一封客户邮件。

背景通常可以包含：

- 当前在做什么
- 任务发生在什么场景
- 内容将给谁看
- 最终用途是什么

示例：

```text
背景：
我正在准备一次部门内部分享，听众没有编程经验。
```

有了这个背景，模型在解释技术内容时，就会更倾向于使用简单语言。

### 6.2 任务

任务需要明确告诉模型要完成什么动作。

尽量使用明确的动词。

| 任务类型 | 常用表达 |
|---|---|
| 总结 | 总结、提炼、概括、压缩 |
| 修改 | 改写、润色、简化、扩写 |
| 分析 | 分析、拆解、判断、归纳 |
| 比较 | 对比、评估、排序 |
| 生成 | 编写、生成、设计、拟定 |
| 提取 | 提取、识别、分类、整理 |
| 检查 | 检查、审查、查找问题 |
| 解释 | 解释、说明、举例 |

例如：

```text
请总结下面的会议记录。
```

```text
请比较以下三个方案。
```

```text
请把下面的内容改写成一封正式邮件。
```

```text
请检查下面的计划是否存在遗漏。
```

任务越明确，模型越容易选择正确的回答方式。

### 6.3 材料

材料是完成任务所需要的信息。

例如：

- 一段文章
- 一份会议记录
- 一封邮件
- 一组数据
- 一个项目介绍
- 一段代码
- 几个候选方案
- 用户反馈
- 当前计划

示例：

```text
材料：
- 项目原计划 6 月完成
- 需求调整了三次
- 测试环境晚交付两周
- 当前开发完成度约为 80%
```

对于整理、总结和分析类任务，材料往往比提示词技巧更重要。

可以把 Chatbot 理解成一个刚加入任务的助手。

如果没有把现有信息告诉它，它就很难生成具体、准确的内容。

### 6.4 要求

要求用于说明结果应该是什么样。

#### 长度

```text
控制在 300 字以内。
```

```text
用三句话概括。
```

```text
生成一份详细版本和一份简洁版本。
```

#### 结构

```text
按照背景、问题、原因和建议四部分组织。
```

```text
先给出结论，再说明理由。
```

```text
使用表格输出。
```

#### 语气

```text
使用正式、简洁的语言。
```

```text
语气礼貌，但不要过于客套。
```

```text
使用容易理解的表达。
```

#### 重点

```text
重点突出项目结果和下一步行动。
```

```text
保留所有数字和时间信息。
```

```text
只保留最重要的三个结论。
```

#### 格式

```text
请使用 Markdown 格式。
```

```text
请输出为检查清单。
```

```text
请使用以下字段：
事项、负责人、截止时间、状态。
```

要求并不是越多越好，只需要写出真正影响使用结果的内容。

### 6.5 补充说明

补充说明主要用于告诉模型如何处理特殊情况。

例如：

```text
缺少的信息请标记为“待补充”。
```

```text
没有明确负责人的事项，请标记为“待确认”。
```

```text
请先给出三个不同方向，再分别说明。
```

```text
如果存在多种理解，请分别列出。
```

这些要求可以帮助模型处理输入中不完整或不明确的部分。

## 7. 通用提示词模板

```text
背景：
[说明当前场景、任务目的或阅读对象]

任务：
请帮我[总结 / 修改 / 分析 / 对比 / 生成 / 检查]……

材料：
[粘贴已有信息或需要处理的内容]

输出要求：
- 内容重点：
- 输出结构：
- 长度：
- 语气：
- 格式：

补充说明：
- [...]
```

示例：

```text
背景：
我正在准备一个项目周会，参会者包括项目成员和部门负责人。

任务：
请把下面的项目记录整理成一份周会汇报。

材料：
- 本周完成了用户登录功能
- 数据导入功能完成约 70%
- 测试环境还没有准备完成
- 当前风险是接口文档更新较慢
- 下周计划完成数据导入并开始联调

输出要求：
- 按“本周进展、当前问题、风险、下周计划”组织
- 使用简洁、正式的语言
- 控制在 400 字以内
- 使用 Markdown 格式
```

## 8. 常见任务一：总结内容

总结是 Chatbot 最常见的使用场景之一。

但“总结”本身可能包含不同目的：

- 快速了解全文
- 提取重要结论
- 提取行动事项
- 为管理层准备摘要
- 为会议准备回顾
- 将长文压缩成短文
- 将非结构化内容整理成表格

因此，在让 Chatbot 总结时，可以说明：

```text
总结给谁看
重点保留什么
需要多长
采用什么结构
```

### 8.1 基础问法

```text
请总结下面的内容。
```

### 8.2 更明确的问法

```text
请总结下面的会议记录。

使用场景：
用于发送会后纪要。

请重点提取：
- 已确认的结论
- 待办事项
- 负责人
- 完成时间
- 仍然存在的分歧

输出要求：
- 先给出会议结论
- 再用表格列出待办事项
- 没有明确负责人或时间的内容标记为“待确认”

会议记录：
[粘贴内容]
```

### 8.3 总结提示词模板

```text
请总结下面的内容。

总结用途：
[...]

阅读对象：
[...]

需要保留：
- [...]
- [...]
- [...]

输出要求：
- 长度：
- 结构：
- 格式：

原文：
[粘贴内容]
```

## 9. 常见任务二：修改和润色

“帮我润色一下”是一个很常见的提示词。

但是“润色”可能代表不同的目标：

- 更正式
- 更简洁
- 更自然
- 更有礼貌
- 更有说服力
- 更适合口头表达
- 更适合管理层阅读
- 更容易被非专业人员理解

因此，修改文字时，最好明确修改方向。

### 9.1 示例

```text
请修改下面这段文字。

使用场景：
发送给其他部门负责人。

修改目标：
- 语气专业、礼貌
- 减少重复表达
- 清楚说明希望对方完成的动作
- 保留原文的主要意思
- 控制在 200 字以内

原文：
[粘贴内容]

请输出：
1. 修改后的版本
2. 主要修改点
```

```text
[...]

修改目标：

- [...]

- [...]

```

## 10. 常见任务三：生成初稿

Chatbot 很适合帮助我们快速生成一个可以继续修改的初稿。

常见内容包括：

- 邮件
- 项目计划
- 会议议程
- 汇报结构
- 工作总结
- 培训材料
- 问题清单
- 采访提纲
- 调研问卷
- 文档目录

生成初稿时，需要说明：

- 内容的使用场景
- 阅读对象
- 目标
- 已知信息
- 期望结构
- 长度和语气

### 10.1 示例

```text
请帮我生成一份项目启动会议议程。

背景：
- 会议时长为 45 分钟
- 参与者包括项目负责人、业务人员和开发人员
- 项目目前只有初步需求

会议目标：
- 确认项目目标
- 明确项目范围
- 确认团队分工
- 确定下一步行动

请输出：
- 议题名称
- 每个议题的建议时长
- 负责人
- 期望产出

总时长不得超过 45 分钟。
```

## 11. 常见任务四：分析问题

分析类任务中，输入信息会明显影响回答质量。

例如：

> 为什么这个项目延期了？

模型只能提供一些常见原因，例如：

- 需求变化
- 资源不足
- 沟通问题
- 技术困难
- 计划不合理

这些回答可能是正确的方向，但不一定和当前项目直接相关。

如果提供更多信息：

```text
项目原计划 6 月完成。

当前情况：
- 需求在开发过程中修改了三次
- 两名核心成员中途被调到其他项目
- 测试环境晚交付两周
- 当前开发完成约 80%
- 暂时没有发现重大技术障碍

请分析项目延期的可能原因。

输出要求：
1. 区分直接原因和间接原因
2. 按影响程度排序
3. 说明每项判断对应的信息
4. 列出下一步需要进一步确认的问题
```

模型就能基于提供的信息进行更具体的分析。

### 11.1 分析提示词模板

```text
[...]

- [...]
```

## 12. 常见任务五：比较方案

让 Chatbot 比较方案时，最好先定义比较标准。

例如：

> 方案 A 和方案 B 哪个更好？

“更好”可能有很多不同含义：

- 成本更低
- 上线更快
- 操作更简单
- 功能更多
- 维护更容易
- 更适合大团队
- 更适合长期使用

因此，可以改成：

```text
请比较方案 A 和方案 B。

使用场景：
一个 5 人团队需要在一个月内完成内部工具的第一版。

比较维度：
- 实施成本
- 开发周期
- 学习成本
- 维护难度
- 后续扩展性

请输出：
1. 对比表
2. 两个方案各自的优点和不足
3. 更适合当前场景的方案
4. 推荐方案成立的条件
```

### 12.1 比较提示词模板

```text
请比较以下选项：
- 方案 A：
- 方案 B：
- 方案 C：

使用场景：
[...]

目标：
[...]

比较维度：
- [...]
- [...]
- [...]

输出要求：
1. 对比表
2. 各方案优缺点
3. 推荐方案
4. 推荐理由
5. 适用条件
```

## 13. 常见任务六：提取和整理信息

Chatbot 可以将一段非结构化文字整理成结构化内容。

例如，一段会议记录中可能同时包含：

- 多个讨论主题
- 不同负责人
- 多个时间节点
- 决策
- 待办事项
- 风险

可以要求模型提取指定字段。

```text
请从下面的会议记录中提取待办事项。

每个待办事项需要包含：
- 事项
- 负责人
- 截止时间
- 当前状态
- 相关背景

请使用表格输出。
没有明确负责人、时间或状态时，填写“待确认”。

会议记录：
[粘贴内容]
```

### 13.1 信息提取模板

```text
请从下面的内容中提取信息。

需要提取的字段：
- [...]
- [...]
- [...]

输出格式：
[表格 / 列表 / JSON]

补充要求：
- 相同内容进行合并
- 保留原文中的数字和时间
- 缺失字段填写“待确认”

原文：
[粘贴内容]
```

## 14. 指定输出形式

有时回答内容本身没有问题，但不方便直接使用。

这时可以通过指定输出形式，让结果更加清晰。

### 14.1 表格

```text
请使用表格输出，字段包括：
问题、影响、原因、建议、优先级。
```

适合：

- 方案对比
- 任务整理
- 问题分析
- 信息提取
- 计划安排

### 14.2 清单

```text
请整理成一份检查清单，每一项使用动词开头。
```

适合：

- 操作步骤
- 准备事项
- 审查内容
- 工作流程

### 14.3 分层结构

```text
请按照以下层级输出：
1. 核心结论
2. 主要依据
3. 具体建议
4. 下一步行动
```

适合：

- 汇报
- 分析
- 决策建议

### 14.4 Markdown

```text
请使用 Markdown 格式输出，包含标题、二级标题和列表。
```

适合：

- 飞书文档
- Wiki
- README
- 技术文档
- 培训材料

### 14.5 两种版本

```text
请分别提供：
1. 一个 100 字以内的简洁版
2. 一个包含完整背景和细节的详细版
```

适合同时满足不同阅读对象。

## 15. 多轮对话：通过追问逐步得到更好的结果

使用 Chatbot 时，不需要把全部要求一次写完。

可以先生成初稿，再根据结果进行修改。

### 第一轮

```text
请帮我写一份项目总结。
```

### 第二轮

```text
这份总结会发给部门负责人，请调整得更加简洁正式。
```

### 第三轮

```text
请重点突出项目成果、当前风险和下一步需要的资源。
```

### 第四轮

```text
请压缩到 500 字以内，并按照“成果、风险、下一步”三个部分组织。
```

通过持续补充要求，模型可以逐步调整输出。

## 16. 常用追问方式

### 16.1 回答太长

```text
请压缩到 300 字以内，只保留最重要的三个结论。
```

### 16.2 回答太简略

```text
请进一步展开第二部分，并增加一个具体示例。
```

### 16.3 结构不清晰

```text
请重新组织结构，先给结论，再说明原因，最后给出建议。
```

### 16.4 内容太泛

```text
请结合我提供的具体信息重新回答，减少通用描述。
```

### 16.5 表达太复杂

```text
请使用更容易理解的语言，减少专业术语，并为必要术语增加解释。
```

### 16.6 希望看到不同方案

```text
请提供三个不同方向的方案，并分别说明各自的特点。
```

### 16.7 希望进一步细化

```text
请把这个方案拆分成具体步骤，并为每一步说明输入、动作和输出。
```

### 16.8 希望调整语气

```text
请保持原意，将语气调整得更礼貌、更自然，不要过于正式。
```

## 17. 让 Chatbot 先帮助我们明确问题

当任务比较复杂时，可以先让 Chatbot 提问。

例如：

```text
我需要制定一个项目计划，但目前的信息还比较零散。

请先不要直接生成计划。
请先向我提出完成这项任务所需要了解的问题。
```

也可以限定问题数量：

```text
请先向我提出 5 个最关键的问题。
在我回答后，再帮我制定方案。
```

这种方式适合：

- 制定项目计划
- 编写重要文档
- 分析复杂问题
- 设计活动
- 制定学习计划
- 准备汇报
- 比较多个选项

## 18. 角色描述有什么作用

提示词中经常会出现：

> 你是一名项目经理。

> 你是一名编辑。

> 你是一名培训讲师。

角色描述可以帮助模型选择一定的表达角度。

例如：

```text
请以项目经理的角度检查下面的计划。
```

模型可能会更加关注：

- 目标
- 时间
- 资源
- 风险
- 负责人
- 里程碑

而如果输入：

```text
请以普通用户的角度检查下面的产品说明。
```

模型可能会更加关注：

- 是否容易理解
- 操作是否清楚
- 术语是否过多
- 使用步骤是否完整

角色描述适合用来规定观察角度，但仅有角色通常还不够。

例如：

```text
你是一名优秀的项目经理，请分析这个项目。
```

仍然没有说明：

- 分析什么
- 使用哪些材料
- 关注哪些维度
- 最后输出什么

更完整的表达是：

```text
请以项目经理的角度审查下面的项目计划。

重点检查：
- 目标是否明确
- 时间安排是否合理
- 是否缺少负责人
- 是否存在依赖关系
- 是否遗漏主要风险

请使用表格输出问题和修改建议。
```

## 19. 示例的重要性

当我们很难用文字描述想要的结果时，可以直接提供一个示例。

例如：

```text
请按照下面的格式整理其他内容。

示例：

事项：完成需求评审
负责人：张三
时间：6 月 10 日
状态：进行中

需要整理的内容：
[粘贴内容]
```

模型会参考示例中的：

- 字段
- 顺序
- 表达方式
- 详细程度
- 格式

这种方法适合：

- 固定格式输出
- 文案改写
- 数据提取
- 分类任务
- 生成相似内容

## 20. 将复杂任务拆成多个步骤

当任务包含多个环节时，可以分步骤进行。

例如，要把一份会议记录整理成项目计划，可以拆成：

```text
第一步：提取会议中的所有待办事项。
```

```text
第二步：为待办事项补充负责人、时间和依赖关系。
```

```text
第三步：按照优先级和时间顺序整理成项目计划。
```

```text
第四步：将项目计划压缩成一页汇报。
```

相比一次要求模型同时完成所有事情，分步骤通常更容易检查和修改。

可以使用以下表达：

```text
请分三个步骤完成这个任务。

第一步：提取关键信息。
第二步：对信息进行分类。
第三步：根据分类结果生成总结。
```

## 21. 一个提示词可以怎样逐步完善

下面以“写会议纪要”为例。

### 第一版

```text

```

### 第二版：明确任务


### 第三版：补充输出内容

```text
- 会议结论
- 讨论事项
- 待办事项
- 负责人
- 截止时间
```

### 第四版：补充格式和规则

```text
请把下面的会议记录整理成会议纪要。

需要包含：
1. 会议结论
2. 讨论事项
3. 待办事项

待办事项请使用表格输出，字段包括：
- 事项
- 负责人
- 截止时间
- 当前状态

```

### 第五版：补充使用场景和表达要求

```text
背景：
这是一场项目周会，纪要将发送给项目成员和部门负责人。

任务：
请把下面的会议记录整理成会议纪要。

需要包含：
1. 核心结论
2. 主要讨论事项
3. 风险和问题
4. 待办事项

输出要求：
- 先给出 200 字以内的会议摘要
- 待办事项使用表格输出
- 表格字段包括事项、负责人、截止时间和状态
- 使用简洁、正式的语言
- 没有明确负责人或时间的内容标记为“待确认”

会议记录：
[粘贴内容]
```

提示词的完善过程，本质上是在逐步减少以下内容的不确定性：

```text
任务是什么
材料是什么
结果给谁使用
需要保留什么
最终怎么展示
```

## 23. 提示词的核心原则

### 原则一：明确任务

不要只描述一个主题，要明确希望模型做什么。

相比：

```text
项目管理。
```

更清楚的是：

```text
请总结项目管理中常见的五类风险。
```

### 原则二：提供上下文

模型不知道没有出现在对话中的信息。

相比：

```text
帮我回复一下。
```

更清楚的是：

```text
这是一封同事询问项目进度的邮件。请帮我回复，说明当前已经完成需求确认，预计周五提供第一版结果。
```

### 原则三：说明使用场景

相同内容面向不同读者，表达方式会不同。

例如：

```text
请分别生成两个版本：
1. 面向管理层的简洁版本
2. 面向项目成员的详细版本
```

### 原则四：明确输出形式

说明结果需要以什么形式呈现。

例如：

```text
请使用表格输出。
```

```text
请整理成一页汇报。
```

```text
请生成一份行动清单。
```

### 原则五：通过追问持续调整

第一次回答可以作为初稿。

之后可以继续要求：

```text
请缩短。
```

```text
请进一步展开。
```

```text
请调整语气。
```

```text
请增加示例。
```

```text
请改成表格。
```

```text
请重新排序。
```

## 24. 总结

提示词并不是一种复杂的编程语言。

它本质上是我们与 Chatbot 沟通任务的方式。

一个清晰的提示词，通常会说明：

```text
背景是什么
需要完成什么
可以使用哪些材料
输出应该是什么样
还有哪些补充要求
```

可以将通用提示词结构记为：

```text
背景 + 任务 + 材料 + 要求 + 补充说明
```

使用 Chatbot 时，也不需要追求一次完成。

更常见的过程是：

```text
提出任务
→ 查看初稿
→ 补充信息
→ 调整结构
→ 修改表达
→ 获得最终结果
```

最终可以将有效使用 Chatbot 概括为：

> **清楚地描述任务，提供必要的信息，明确输出要求，并通过对话持续调整。**
